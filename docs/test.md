### Day 1: 系统架构宏观视角

#### 🤔 思考题 1：为什么 Controller 和 Agent 都有 `master_thread` 和 `slave_thread`？它们职责有何不同？

**参考答案：**

**职责划分：**

| 组件 | master_thread | slave_thread |
|------|---------------|--------------|
| **Controller** | ✓ 主控制逻辑<br>✓ 任务调度<br>✓ 全局决策 | ✗ 不存在 |
| **Agent** | ✗ 不存在 | ✓ 本地执行<br>✓ 事件处理<br>✓ 消息转发 |

**设计原因：**

1. **对称性考虑**：
   - Controller 需要集中管理多个 Agent，因此使用 `master_thread` 作为"大脑"
   - Agent 需要响应 Controller 指令并管理本地资源，因此使用 `slave_thread` 作为"执行者"
   - 这种命名反映了主从关系（Master-Slave pattern）

2. **代码复用**：
   ```cpp
   // controller/src/beerocks/master/master_thread.cpp
   class master_thread {
       task_pool tasks;  // 管理所有 Task
       db database;      // 全局数据库
   };
   
   // agent/src/beerocks/slave/son_slave_thread.cpp  
   class slave_thread {
       task_pool tasks;  // 本地 Task
       AgentDB database; // 本地数据库
   };
   ```

3. **实际差异**：
   - Controller 的 master_thread 初始化更多全局管理 Task（如 topology_task, client_steering_task）
   - Agent 的 slave_thread 初始化本地执行 Task（如 ap_autoconfiguration_task, backhaul_manager）

**查看代码验证：**
```bash
# Controller 初始化的任务
grep "add_task" controller/src/beerocks/master/master_thread.cpp
  → topology_task
  → agent_monitoring_task
  → bml_task

# Agent 初始化的任务  
grep "add_task" agent/src/beerocks/slave/son_slave_thread.cpp
  → TopologyTask
  → ApAutoConfigurationTask
  → BackhaulManager
```

**参考答案：**

**Framework 层提供的服务：**

```
framework/
├── common/           # 公共工具库
│   ├── logger.cpp    # 日志系统 (easylogging++)
│   ├── utils.cpp     # 字符串、时间等工具
│   └── encryption.cpp # 加密工具 (HMAC)
│
├── transport/        # 传输层
│   └── ieee1905_transport/
│       ├── socket_connection.h  # UDP Socket 封装
│       ├── cmdu_tx_rx.h         # CMDU 编解码
│       └── broker_server.cpp    # 消息代理服务器
│
├── tlvf/             # TLV 框架
│   ├── tlvf.py       # 代码生成器
│   ├── yaml/tlvf/    # YAML 定义文件
│   └── AutoGenerated/ # 生成的 C++ 代码
│
├── platform/         # 平台抽象层
│   ├── bpl/          # Board Platform Layer
│   │   ├── bpl_cfg.h # 配置接口
│   │   ├── bpl_wlan.h # WiFi 接口
│   │   └── bpl_dcs.h # DCS 接口
│   ├── nbapi/        # North Bound API (ODL 接口)
│   └── wbapi/        # West Bound API
│
└── external/         # 外部依赖
    ├── easylogging/  # 日志库
    └── enumAutoPrint/ # 枚举打印工具
```

**为什么要独立出 Framework 层？**

1. **分层解耦**：
   - 上层不依赖下层具体实现
   - 便于移植到不同平台

2. **代码复用**：
   - Controller 和 Agent 都使用相同的 TLV 编解码
   - 都使用相同的 IEEE 1905 传输层
   - 避免重复实现

3. **可测试性**：
   - Framework 层可以独立单元测试
   - Mock 平台层进行仿真测试

4. **可扩展性**：
   - 新增平台只需实现 BPL 接口
   - 不影响上层业务逻辑

#### 💡 产品思考环节

**Q1: EasyMesh 协议定义了 Controller 和 Agent 两种角色，这解决了什么用户痛点？**

**参考答案：**

**解决的痛点：**

| 痛点 | 传统方案 | EasyMesh 方案 |
|------|----------|---------------|
| **多厂商兼容** | 各厂商私有协议，无法混用 | 标准化接口，A 厂 Controller + B 厂 Agent |
| **集中管理** | 每台设备独立配置 | 一台 Controller 统一管理所有 Agent |
| **智能决策** | 分布式决策，效率低 | Controller 全局视角，优化决策 |
| **即插即用** | 手动配置每个节点 | Agent 自动发现 Controller，零配置 |

**用户价值：**
- **降低复杂度**：用户只需配置 Controller，Agent 自动同步
- **提升体验**：无缝漫游、自动优化
- **降低成本**：多厂商竞争，避免绑定

---

**Q2: 如果让你设计，会把哪些功能放在 Controller，哪些放在 Agent？**

**参考答案：**

**设计原则：**
- **集中决策** vs **分布执行**
- **全局优化** vs **本地快速响应**

**功能分配建议：**

| 功能 | 位置 | 理由 |
|------|------|------|
| 拓扑发现与维护 | Controller | 需要全局视图 |
| 漫游决策 | Controller | 跨 AP 协调 |
| 信道选择 | Controller | 避免邻区干扰 |
| 负载均衡 | Controller | 全局资源分配 |
| AP 配置执行 | Agent | 本地操作 |
| 回程管理 | Agent | 快速响应链路变化 |
| 客户端关联 | Agent | 实时性要求高 |
| 指标采集 | Agent | 本地测量 |

**我的设计改进：**
```
引入"分层决策"：
- 战略决策（漫游、信道）→ Controller
- 战术决策（关联、重传）→ Agent
- 支持离线模式：Agent 在 Controller 失联时自主运行
```

   - 高端产品：同时支持

### Day 2: 启动流程与状态机雏形

#### 🤔 思考题 1：从 `INIT` 到 `OPERATIONAL`，中间经历了哪些必要状态？为什么不能直接跳转？

**参考答案：**

**完整状态转换路径：**

```
有线回程场景：
INIT → WAIT_ENABLE → ENABLED → CONNECTED → OPERATIONAL

无线回程场景：
INIT → WAIT_ENABLE → ENABLED → INIT_HAL → WPA_ATTACH → 
INITIATE_SCAN → WAIT_FOR_SCAN_RESULTS → WIRELESS_CONFIG_4ADDR_MODE → 
WIRELESS_ASSOCIATE_4ADDR → WIRELESS_ASSOCIATE_4ADDR_WAIT → 
CONNECTED → OPERATIONAL
```

**为什么不能跳过？**

| 状态 | 必要性 | 跳过后果 |
|------|--------|----------|
| **WAIT_ENABLE** | ✓ 等待 Slave 准备就绪 | 可能访问未初始化资源 |
| **ENABLED** | ✓ 确认配置已加载 | 配置不完整导致错误 |
| **INIT_HAL** (无线) | ✓ 初始化 WiFi HAL | HAL 未初始化，无法扫描 |
| **WPA_ATTACH** (无线) | ✓ 连接 wpa_supplicant | 无法管理 WiFi |
| **INITIATE_SCAN** (无线) | ✓ 发现可用 AP | 不知道连接哪个 |
| **CONFIG_4ADDR** (无线) | ✓ 启用 4 地址模式 | 数据包无法正确转发 |

**设计哲学：**
- **显式状态** > 隐式假设
- **逐步验证** > 一步到位
- **失败回滚** > 崩溃退出

#### 🤔 思考题 2：`RESTART` 状态的设计目的是什么？如果连续重启失败会怎样？

**参考答案：**

**RESTART 状态的作用：**

```cpp
case EState::RESTART: {
    // 1. 清理资源
    for (auto &radio_info : m_radios_info) {
        clear_radio_handlers(radio_info);
        if (radio_info->sta_wlan_hal) {
            radio_info->sta_wlan_hal.reset();
        }
    }
    
    // 2. 通知断开
    handle_backhaul_disconnect();
    
    // 3. 检查重试次数
    if (configuration_stop_on_failure_attempts && !stop_on_failure_attempts) {
        platform_notify_error(bpl::eErrorCode::BH_STOPPED, "backhaul manager stopped");
        FSM_MOVE_STATE(STOPPED);  // 永久停止
    } else {
        FSM_MOVE_STATE(INIT);  // 重新初始化
    }
}
```

**设计目的：**

1. **优雅降级**：不是直接 crash，而是有序清理
2. **自愈机制**：尝试重新启动
3. **防止无限重启**：限制重试次数

**连续失败场景：**
```
第 1 次失败：RESTART → INIT (重试)
第 2 次失败：RESTART → INIT (重试)
第 3 次失败：RESTART → STOPPED (永久停止)
```

**产品意义：**
- **可维护性**：运维人员收到告警
- **安全性**：避免设备反复重启影响网络
- **可诊断**：日志记录失败原因

### Day 3: 拓扑发现机制

#### 🤔 思考题 1：为什么拓扑发现周期是 60 秒？太短或太长会有什么问题？

**参考答案：**

**IEEE 1905.1 标准规定：**

```cpp
constexpr uint8_t TOPOLOGY_DISCOVERY_TX_CYCLE_SEC = 60;
```

**60 秒的权衡：**

| 周期 | 每分钟消息数 (10 个 Agent) | 平均发现延迟 | 评价 |
|------|---------------------------|--------------|------|
| **10 秒** | 60 次/分 | 5 秒 | ❌ 太频繁 |
| **60 秒** | 10 次/分 | 30 秒 | ✓ 平衡点 |
| **300 秒** | 2 次/分 | 150 秒 | ❌ 太慢 |

**太短的问题 (<30 秒)：**
1. **网络拥塞**：Discovery 消息占用带宽
2. **CPU 负载**：频繁处理消息
3. **误判风险**：临时丢包被认为离线

**太长的问题 (>120 秒)：**
1. **用户体验差**：设备离线很久才发现
2. **漫游失败**：拓扑未及时更新
3. **配置延迟**：新 Agent 加入后长时间无法配置

**补充机制：**
```cpp
// 事件触发补充
if (agent_connected || agent_disconnected) {
    send_topology_notification();  // 立即通知，不用等 60 秒
}
```

#### 🤔 思考题 2：`TOPOLOGY_DISCOVERY` 和 `TOPOLOGY_QUERY` 有什么区别？

**参考答案：**

**消息对比：**

| 特性 | TOPOLOGY_DISCOVERY | TOPOLOGY_QUERY |
|------|-------------------|----------------|
| **发起方** | Agent (周期性) | Controller (按需) |
| **目的** | "我在这里！" | "告诉我你的详细信息" |
| **内容** | 简单：AL MAC + Device MAC | 详细：Radio 信息、BSS 列表等 |
| **周期** | 60 秒 | 事件触发 |
| **传输方式** | 多播 | 单播 |

**交互流程：**
```
Agent --Discovery(多播)--> Controller
                            ↓
                     Controller 发现新设备
                            ↓
         Controller --Query(单播)--> Agent
                            ↓
              Agent --Response(单播)--> Controller
                            ↓
                    Controller 更新 Database
```

**为什么要两种消息？**

1. **效率考虑**：
   - Discovery 简单，用于快速发现
   - Query/Response 复杂，只在需要时发送

2. **网络负载**：
   - Discovery 多播：所有 Controller 都能听到
   - Query/Response 单播：减少无关设备处理

3. **灵活性**：
   - Controller 可以主动查询特定 Agent
   - 不必等待 60 秒周期

### Day 4: 设备发现与握手

#### 🤔 思考题 1：是 Agent 发现 Controller，还是 Controller 发现 Agent？

**参考答案：**

**双向发现机制：**

```
阶段 1: Agent 发现 Controller
┌────────┐  Discovery(多播) ┌────────────┐
│ Agent  │ ───────────────→ │ Controller │
└────────┘                  └────────────┘
                                  ↓
                            记录新 Agent
                                  ↓
                             发送 Query
                                  ↓
┌────────┐ ←─────────────── ┌────────────┐
│ Agent  │ Response(单播)   │ Controller │
└────────┘                  └────────────┘

阶段 2: Controller 监控 Agent
┌────────┐  定期 Discovery   ┌────────────┐
│ Agent  │ ───────────────→ │ Controller │
└────────┘                  └────────────┘
     ↑                          ↓
     └──────────────────  超时判断离线
```

**代码验证：**

```cpp
// Agent 侧：主动发送 Discovery
void TopologyTask::send_topology_discovery() {
    auto cmdu_header = m_cmdu_tx.create(0, 
        ieee1905_1::eMessageType::TOPOLOGY_DISCOVERY_MESSAGE);
    // 添加 AL MAC TLV
    auto tlvAlMacAddress = m_cmdu_tx.addClass<ieee1905_1::tlvAlMacAddress>();
}

// Controller 侧：接收并响应
void topology_task::handle_topology_discovery(src_mac, cmdu_rx) {
    if (!database.m_agents.get(al_mac)) {
        // 新 Agent，发送 Query
        son_actions::send_topology_query_msg(al_mac, cmdu_tx, database);
    }
}
```

**结论：**
- **第一阶段**：Agent 主动发现 Controller（通过多播 Discovery）
- **第二阶段**：Controller 反向确认 Agent（通过 Query）
- **持续监控**：双向心跳检测

#### 🤔 思考题 1：M1、M2、M8 分别是什么消息？谁发给谁？

**参考答案：**

**WSC 消息类型：**

```
M1 (Message 1):
┌────────┐                         ┌────────────┐
│ Agent  │ ───── M1 Request ────→ │ Controller │
│ (Enrollee) │                     │ (Registrar) │
└────────┘                         └────────────┘
内容：Enrollee 能力、公钥

M2 (Message 2):
┌────────┐                         ┌────────────┐
│ Agent  │ ←──── M2 Response ──── │ Controller │
│ (Enrollee) │                     │ (Registrar) │
└────────┘                         └────────────┘
内容：SSID、密码、认证方式、加密类型

M8 (Message 8):
┌────────┐                         ┌────────────┐
│ Agent  │ ←──── M8 Final ─────── │ Controller │
│ (Enrollee) │                     │ (Registrar) │
└────────┘                         └────────────┘
内容：bSTA 配置（可选，用于回程 STA）
```

**代码中的解析：**

```cpp
// ap_autoconfiguration_task.cpp
for (auto tlv : cmdu_rx.getClassList<ieee1905_1::tlvWsc>()) {
    WSC::WscAttrList::wsc_header *tlv_header = 
        reinterpret_cast<WSC::WscAttrList::wsc_header *>(tlv->payload());
    
    if (tlv_header->msg_type_value == WSC::eWscMessageType::WSC_MSG_TYPE_M2) {
        m2 = WSC::m2::parse(*tlv);  // 解析 M2
        handle_wsc_m2_tlv(..., bss_infos);
    } else if (tlv_header->msg_type_value == WSC::eWscMessageType::WSC_MSG_TYPE_M8) {
        m8 = WSC::m8::parse(*tlv);  // 解析 M8
        handle_wsc_m8_tlv(..., bsta_info);
    }
}
```

**记忆技巧：**
- M1：我要入网（Enrollee 请求）
- M2：给你配置（Registrar 响应）
- M8：最终确认（可选）

### Day 8: 无线回程建立

#### 🤔 思考题 1：什么是 4 地址模式？为什么无线回程需要它？

**参考答案：**

**4 地址模式详解：**

**普通 WiFi 帧（3 地址）：**
```
┌──────────┬──────────┬──────────┐
│   DA     │   SA     │   BSSID  │  ← 只有 3 个 MAC 地址
│ (目标)   │ (源)     │ (AP)     │
└──────────┴──────────┴──────────┘
```

**4 地址模式（WDS）：**
```
┌──────────┬──────────┬──────────┬──────────┐
│   DA     │   SA     │   TA     │   RA     │
│ (目标)   │ (源)     │ (发送 AP)│ (接收 AP)│
└──────────┴──────────┴──────────┴──────────┘
```

**为什么需要 4 地址？**

**场景：无线回程**
```
[手机] --WiFi--→ [子路由] --无线回程--→ [主路由] --WAN--→ 互联网

如果没有 4 地址：
手机 MAC: AA:AA:AA:AA:AA:AA
子路由 MAC: BB:BB:BB:BB:BB:BB
主路由看到：源 MAC = BB:BB:BB:BB:BB:BB (丢失了手机 MAC)

有 4 地址：
DA=主路由，SA=手机，TA=子路由，RA=主路由
主路由能看到原始手机 MAC
```

**代码中的设置：**
```cpp
// backhaul_manager.cpp
case EState::WIRELESS_CONFIG_4ADDR_MODE: {
    active_hal->disconnect();  // 先断开
    active_hal->set_4addr_mode(true);  // 启用 4 地址
    FSM_MOVE_STATE(WIRELESS_ASSOCIATE_4ADDR);
}
```

**类比理解：**
- 3 地址：邮局中转（只看到邮局地址）
- 4 地址：快递面单（发件人 + 收件人都清晰）

### Day 10: 漫游引导

#### 🤔 思考题 1：Controller 如何判断一个 STA 需要漫游？

**参考答案：**

**漫游决策算法：**

```cpp
// optimal_path_task.cpp
void optimal_path_task::work() {
    // 1. 收集 RSSI 信息
    for (auto &station : m_database.m_stations) {
        int rx_rssi = station.second.rx_rssi;
        
        // 2. 计算最优路径
        if (rx_rssi < m_database.config.roaming_rssi_cutoff_db) {
            // 信号低于阈值，考虑漫游
            find_better_ap(station);
        }
        
        // 3. 评估候选 AP
        for (auto &candidate : candidate_aps) {
            if (candidate.rssi > current.rssi + HYSTERESIS_DB) {
                // 候选 AP 信号更好（有迟滞）
                trigger_roaming(station, candidate);
            }
        }
    }
}
```

**关键指标：**

| 指标 | 阈值 | 说明 |
|------|------|------|
| **RSSI 阈值** | -75dBm | 低于此值考虑漫游 |
| **迟滞** | 10-15dB | 避免乒乓效应 |
| **信噪比** | SNR > 25dB | 质量指标 |
| **负载均衡** | STA 数 < 阈值 | 避免过载 |

**产品设计考虑：**

```
漫游策略可配置：
[性能优先]   RSSI < -75dBm → 立即漫游
[稳定优先]   RSSI < -80dBm 且 持续 5 秒 → 漫游
[激进策略]   RSSI < -70dBm → 漫游
[保守策略]   RSSI < -85dBm → 漫游
```

#### 🤔 思考题 2：BTM vs Deauth，如何选择？

**参考答案：**

**两种方式对比：**

| 特性 | BTM (802.11v) | Deauth (强制) |
|------|---------------|---------------|
| **标准** | 802.11v | 802.11 (基础) |
| **兼容性** | 仅支持 11v 的设备 | 所有设备 |
| **体验** | 优雅、不断连 | 粗暴、短暂断连 |
| **成功率** | 依赖 STA 配合 | 强制生效 |
| **时延** | ~100ms | ~500ms |

**代码中的选择逻辑：**

```cpp
// son_slave_thread.cpp
bool slave_thread::handle_client_steering_request(cmdu_rx) {
    auto associated_sta = radio->associated_clients.find(sta_mac);
    
    // 检查是否支持 802.11v
    if (associated_sta->second.supports_11v && 
        !btm_steering_disallowed &&
        db->device_conf.send_btm_to_non_11v_sta) {
        
        // 使用 BTM
        auto request = message_com::create_vs_message<
            beerocks_message::cACTION_APMANAGER_CLIENT_BSS_STEER_REQUEST>(cmdu_tx);
        // 发送 BTM Request
        
    } else {
        // 使用 Deauth 强制断开
        auto request = message_com::create_vs_message<
            beerocks_message::cACTION_APMANAGER_CLIENT_DISCONNECT_REQUEST>(cmdu_tx);
        request->type = beerocks_message::eDisconnect_Type_Deauth;
    }
}
```

**最佳实践：**

```
优先级顺序：
1. 首选 BTM (如果 STA 支持 11v)
2. 次选 Deauth (不支持 11v 或 BTM 失败)
3. 黑名单辅助 (阻止原 AP，迫使 STA 漫游)

产品设计：
- 提供"漫游激进程度"配置
- 记录 BTM 成功率，动态调整策略
```

### Day 15: DPP 安全配置

#### 🤔 思考题 1：DPP 如何解决安全问题？

**参考答案：**

**DPP (Device Provisioning Protocol) 原理：**

**传统 WPS 的安全问题：**
```
WPS PIN 码爆破攻击：
- 8 位 PIN 码 → 10^8 种可能
- 但校验分两段 → 实际只有 10^4 + 10^4
- 暴力破解只需几小时
```

**DPP 的改进：**

```
DPP 流程：
┌────────┐                          ┌────────────┐
│ 手机    │                          │ 新设备     │
│ (Initiator) │                      │ (Enrollee) │
└───┬────┘                          └────┬───────┘
    │                                    │
    │ 1. 扫描二维码                       │
    │    (包含公钥 + 设备信息)             │
    │                                    │
    │ 2. DPP 认证交换                     │
    │    (ECDH 密钥协商)                   │
    │                                    │
    │ 3. 安全配置下发                     │
    │    (加密的 WiFi 凭证)                │
    │                                    │
    └────────────────────────────────────┘
```

**安全特性：**

| 特性 | WPS | DPP |
|------|-----|-----|
| **密钥长度** | 8 位 PIN | 256 位 ECC |
| **抗暴力破解** | ❌ 弱 | ✓ 强 |
| **中间人攻击** | ❌ 易受攻击 | ✓ 有防护 |
| **用户体验** | 按按钮/输 PIN | 扫码 |

**代码中的 DPP 支持：**

```cpp
// 查找 DPP 相关代码
grep -r "DPP" agent/src/beerocks/slave/tasks/ap_autoconfiguration_task.cpp

// DPP onboarding 流程

    // 1. 解析 DPP URI (二维码)
    // 2. DPP 发现阶段
    // 3. DPP 认证阶段
    // 4. DPP 配置阶段
}
```

**产品价值：**

1. **安全性**：量子安全级别的加密
2. **便捷性**：一扫即连
3. **可视化**：二维码可印刷在产品上
4. **防蹭网**：无需暴露密码

```
┌─────────────────────────────────────────┐
│          复杂性维度                      │

│    - 单频/双频/三频                      │
│    - 家庭/企业场景                       │
│                                       │
│ 4. 安全性要求                           │
│    - WPA3、DPP                          │
│    - 防攻击、防篡改                      │
│                                       │
│ 5. 性能优化                             │
│    - 快速漫游 (<50ms)                   │
│    - 负载均衡                           │
│    - 信道优化                           │
└─────────────────────────────────────────┘
```

**能否简化？**

**可以简化的部分：**
1. **配置简化**：
   - 默认配置模板
   - 一键优化

2. **部署简化**：
   - 云端管理
   - App 可视化配置

3. **调试简化**：
   - 统一日志格式
   - 远程诊断

**难以简化的部分：**
1. **核心协议**：互操作性要求

3. **性能优化**：用户体验刚需

**我的简化建议：**

```
┌─────────────────┐

```
